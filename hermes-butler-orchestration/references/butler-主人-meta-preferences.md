# 主人对管家工作流的元偏好（从公文写作会话沉淀）

> 适用：管家/小糖 在派子 agent 与自己动手之间的边界判断。
> 来源：2026-06-03 黔西南州院半年报告写作期间，主人多次打断子 agent 流程的反馈。

## 硬规则（主人原话级别）

### 1. "你改了个寂寞" / "你改了什么？"

**症状**：子 agent 第 4 轮反复 patch 同一份 docx，段落序号跳号、插入位置错位、
小标题残缺——但子 agent 报告"已修复"。
**主人原话**："你改了个寂寞" / "你改了什么？我看不出来"。

**管家动作**：
- 看到子 agent 第 3 轮 patch 失败 → **立刻停止子 agent 循环**
- 改用 python-docx 整段重写（参考 `gongwen-pro-cur/references/half-year-pitfalls.md`）
- 写完跑结构验证脚本（`gongwen-pro-cur/scripts/structure_validator.py`）

### 2. "别启动子agent了，你直接修改完善"

**触发场景**：写公文、写正式报告、写 10% 精修/收尾这类**需要文风一致性**的任务。
**主人原话**："别启动子agent了，你直接修改完善"。

**管家动作**：
- 写 / 改 / 润色 / 校对 → 管家亲自做
- 派子 agent 只做**纯读取**类工作（提取数据、统计关键词、列文件清单）
- 派子 agent 写出来的东西**只是初稿**——管家必须**全部重读 + 改写**

### 3. "不用写，只需要你记住这个风格"

**触发场景**：主人说"分析任务"/"记住这个风格"/"沉淀偏好"——**不是要立即产出**。
**主人原话**："不用写，只需要你记住这个风格" / "沉淀到 skill 里"。

**管家动作**：
- **不要**立即开写
- 把"风格/偏好/规则"沉淀到 skill（最优先）或 memory
- 完成后向主人汇报"已沉淀到 X skill / Y references"

### 4. "同类项目应先 find 终稿0X.docx"

**触发场景**：同类任务重复出现（半年报 → 年度汇报；第一次专班 → 第二次专班）。
**主人原话**："同类项目（半年报/年度汇报）应先 find 主人已交付的'终稿0X.docx'对比学习再动笔"。

**管家动作**：
- 收到同类任务**第一件事**：`find` 主人已交付的"终稿*"参考文件
- 提取文风、结构、口径，与主人已确认的样本对齐
- 写之前告诉主人"已参考终稿0X，准备按 N 段式写"

## 软偏好

### A. 主人倾向"积极姿态"而非"被动应付"

- 写"扛牢试点责任，举全州之力做实"✓
- 写"建议加强、加快推进" ✗（太空）

### B. 主人倾向"摆事实"而非"摆评价"

- 写"应用反馈少则迭代优化缺依据" ✓（具体问题）
- 写"全国试点成色不足" ✗（评价性话）

### C. 主人倾向"温和批评"而非"重锤"

- 写"优势未能充分发挥" ✓
- 写"严重不足、亟需整改" ✗

### D. 主人倾向"自我定位精准"

- 写"立足全国研发院与试点院定位" ✓
- 写"核心研发单位" ✗（自造身份）

## 派子 agent 的安全区间

**可以派（纯读取/纯计算）**：
- 提取 docx/txt/xlsx 内容
- 统计关键词频次
- 跨文件交叉验证数字
- 列文件清单、读 PDF

**绝对不派**：
- 写正式文稿（公文、报告、致辞）
- 写 10% 精修 / 收尾
- 改写主人已交付的文风样本
- 处理敏感信息（检察系统工作内容）

**灰色地带（派但要管家复审）**：
- 写初稿 → 管家必须改写
- 改格式 → 跑完脚本管家必须 spot-check
- 数据分析 → 管家必须自己跑一遍验证
